# ADR-003: Diff-Based Validation Gate Using `examples/phase-2/{before,after}/`

## Status

Accepted

## Date

2026-05-10

## Context

Each rollout release (per ADR-001) needs an objective gate that says "this release is ready to merge." Three measurement strategies were considered:

1. **Diff-based**: Run the new pipeline over each `examples/phase-2/before/<id>.docx`; compare the result to `examples/phase-2/after/<id>.docx`. The release passes when, for the tags in this release's scope, the diff is empty across all 10 pairs.
2. **Time-based**: The user manually times rework on a sample of articles before and after each release. Improvement ratio is the gate.
3. **Downstream-success-based**: Run each `before/<id>.docx` through the SciELO Markup plugin manually and count articles that survive without warnings.

The corpus is fixed: 10 article pairs (`5136, 5293, 5313, 5419, 5424, 5434, 5449, 5458, 5523, 5549`). Article `5548` exists in `examples/formatted/` but not in `examples/phase-2/`; it is excluded.

## Decision

Adopt **diff-based validation** as the primary release gate. For each release:

1. Run the Phase 2 pipeline over each `examples/phase-2/before/<id>.docx`.
2. Compare to `examples/phase-2/after/<id>.docx`, scoped to the tags introduced by the current release plus all tags from previous releases.
3. The release passes when **all 10 pairs match** within the cumulative scope.

The diff scope is incremental:
- Phase 1 (Stage 1 author fix) gate: the resulting `.docx` matches what Markup expects (validated by re-running Markup on 5313/5449 and confirming author auto-marking succeeds).
- Phase 2 gate: scope = `[elocation]`, `[kwdgrp]`/`[kwd]`, `[abstract]`.
- Phase 3 gate: scope = Phase 2 tags + author-block xrefs (`[corresp]`, `xref ref-type="aff"`, `xref ref-type="corresp"`, `[authorid]`, author attributes).
- Phase 4 gate: scope = Phase 2 + Phase 3 tags + `[hist]` with `received`/`accepted`/`histdate datetype="pub"`.

If the diff fails on any pair, the release does not promote. Either the rule is fixed, the corpus pair is amended (with justification), or the rule is descoped.

## Alternatives Considered

### Alternative — Time-Based Measurement

- **Description**: User times manual rework before/after each release on a sample article.
- **Pros**: Closest to the actual pain (operator time saved).
- **Cons**: Tacit, non-automatable, susceptible to confounders (operator fatigue, article complexity). Cannot run in CI; cannot detect regressions. Difficult to reproduce.
- **Why rejected**: Cannot serve as a gate; useful only as a qualitative sanity check.

### Alternative — Downstream-Success-Based

- **Description**: Run each candidate `.docx` through SciELO Markup manually and count clean passes.
- **Pros**: Measures true downstream effect.
- **Cons**: Requires Windows + Word + Markup installed for every gate run; manual; noisy (Markup logs warnings for reasons unrelated to Phase 2 scope).
- **Why rejected**: Operationally impractical and signal-quality is worse than diffing against a known-good `after` corpus.

## Consequences

### Positive

- Automatable and reproducible. Can run in CI.
- The gate is the same artifact that defines the spec: `after/X.docx` IS the truth for what the pipeline must produce.
- Regression detection is automatic — once a Phase passes, subsequent Phases that break it surface immediately.

### Negative

- Trusts the curated `after/` corpus to be correct. If a sample is mislabeled, the gate will mislead; the diff result is only as good as the corpus.
- The diff comparison must be tag-aware (whitespace and unrelated metadata changes must not produce false positives). This is a non-trivial implementation detail deferred to TechSpec.
- Sample size of 10 cannot statistically prove no regression on the long tail; it can only prove correctness on the curated set.

### Risks

- **Risk**: An article in `after/` was hand-edited and contains markup the rule cannot reproduce because it depended on operator judgment (e.g., a non-standard heading).
  **Mitigation**: When such a case appears, document the divergence in the article's row of a tracking note and either descope or refine the rule.
- **Risk**: The corpus stays small (10) and a real-world article reveals a pattern not represented.
  **Mitigation**: Add new before/after pairs to the corpus over time; treat each addition as a corpus update, not a release blocker.

## Implementation Notes

- Diff comparison is tag-scoped (compare presence and attributes of tags within the current scope; ignore tags outside scope and ignore unrelated whitespace).
- A small CLI utility (e.g., `docformatter phase2-verify`) can be added later to automate the gate; deferred to TechSpec.
- The gate is a release condition, not a per-commit check — running it on every commit would be expensive (10 .docx files × pipeline run). Consider running it pre-merge or as a manual check at release tag.
