# ADR-005: .NET 10 LTS with TreatWarningsAsErrors, overriding the spec's .NET 8 default

## Status

Accepted

## Date

2026-05-06

## Context

The original DocFormatter spec (`instructions.md`) locks the runtime to **.NET 8 LTS**. The justification was conservative LTS targeting for a Windows 10 portable `.exe`. Since the spec was written, **.NET 10 LTS** has shipped (November 2025) and is now the current LTS release; .NET 9 was a Standard Term Support release whose support window has ended.

Independently, the user requested that the build treat compiler warnings as errors so unreviewed warnings cannot accumulate. The spec does not address this dimension.

Both changes touch the build configuration and apply uniformly across all three projects (`Core`, `Cli`, `Tests`).

## Decision

The MVP targets **`net10.0`** for all three projects. `Directory.Build.props` at the solution root sets the following properties for every csproj:

```xml
<PropertyGroup>
  <TargetFramework>net10.0</TargetFramework>
  <LangVersion>latest</LangVersion>
  <Nullable>enable</Nullable>
  <ImplicitUsings>enable</ImplicitUsings>
  <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
  <EnforceCodeStyleInBuild>true</EnforceCodeStyleInBuild>
</PropertyGroup>
```

The publish command remains the spec's recipe with the framework swapped:

```
dotnet publish DocFormatter.Cli -c Release -r win-x64 \
  --self-contained true \
  -p:PublishSingleFile=true \
  -p:IncludeNativeLibrariesForSelfExtract=true \
  -p:PublishReadyToRun=true \
  -p:PublishTrimmed=false
```

`PublishTrimmed=false` is still mandatory because `DocumentFormat.OpenXml` uses reflection.

## Alternatives Considered

### Alternative 1: Keep .NET 8 LTS as the spec dictates

- **Description**: Match the spec verbatim. .NET 8 is still in active LTS support until November 2026.
- **Pros**: Zero divergence from the existing written spec; one less ADR.
- **Cons**: New code on a soon-to-age runtime; misses 18 months of language and runtime improvements (collection expressions, `params` collections, `lock` object refinements, AOT improvements). Forces a framework migration before LTS rollover.
- **Why rejected**: The user explicitly asked for .NET 9 or 10. .NET 10 is the active LTS — picking it now avoids a migration during Phase 2.

### Alternative 2: .NET 9 (Standard Term Support)

- **Description**: Use .NET 9 as the user listed it as an option.
- **Pros**: Minimal version bump from the spec.
- **Cons**: STS releases get 18 months of support and .NET 9's window has effectively closed by mid-2026. Picking STS over LTS for a tool with multi-year operational life is a known anti-pattern.
- **Why rejected**: LTS posture is more appropriate for a tool used by editors over multiple journal cycles.

### Alternative 3: Per-project warning policy instead of solution-wide

- **Description**: Set `TreatWarningsAsErrors` only on `DocFormatter.Core`, leave `Cli` and `Tests` warning-only.
- **Pros**: Test fixtures and CLI scaffolding sometimes generate noisy warnings (unused parameter, async without await) that are tolerable in non-library code.
- **Cons**: Mixed policies create gray zones — a warning in `Cli` could mask a real issue. Test code is where bugs hide, so warnings there are not less valuable.
- **Why rejected**: Solution-wide is simpler and stricter. Specific warnings can be silenced per-project with `<NoWarn>` if a real false positive emerges.

## Consequences

### Positive

- The MVP runs on the current LTS; no migration needed during Phase 2 or Phase 3.
- Warnings cannot accumulate silently; every warning is a build break that forces review.
- Access to .NET 10 language and runtime features (collection expressions, `params` collections, etc.) for cleaner rule code.

### Negative

- The original spec is now the source of two overrides (this ADR + ADR-003 for ORCID). Future contributors must read the ADRs to know what's authoritative.
- Build will fail on warnings the spec author did not anticipate (e.g., nullability warnings on OpenXML SDK return types).

### Risks

- **Risk**: A third-party NuGet package hasn't shipped a .NET 10-targeted assembly yet and forces a downlevel reference.
  - **Mitigation**: All MVP dependencies (`DocumentFormat.OpenXml` 3.x, `Microsoft.Extensions.DependencyInjection`, `Serilog`, `xunit`) target `netstandard2.0` or `net8.0+`, all of which are consumed transparently by `net10.0`.
- **Risk**: A genuine compiler warning blocks the MVP delivery while debugging.
  - **Mitigation**: `<NoWarn>` per-project or per-file `#pragma warning disable` are accepted escape hatches, but each use must be justified in a code comment naming the warning code. Code review enforces the justification.
- **Risk**: Windows 10 users without the .NET 10 runtime cannot run a non-self-contained build.
  - **Mitigation**: Spec already mandates `--self-contained true` + `PublishSingleFile=true`. The portable `.exe` carries the runtime; the user does not install .NET. This ADR does not change the publish recipe.

## Implementation Notes

- `global.json` is **not** added to the solution. Pinning the SDK forces a specific minor version on contributors and CI; for a single-developer MVP it is unnecessary friction. The `TargetFramework` property in `Directory.Build.props` already locks the runtime.
- If any rule needs to use the new collection expression syntax (`[..]`), that's allowed; `LangVersion=latest` is set.
- Compiler warning codes specifically expected (and to be addressed, not silenced):
  - `CS8600`/`CS8602` (nullability) — fix by annotating types or guarding with `??`/`is not null`.
  - `CS8625` (null literal to non-nullable) — fix by declaring the field `?`.
  - `CA1822` (member can be static) — fix by making the method static.
