# ADR-001: Rollout Strategy — Help SciELO Markup, Don't Replace It

## Status

Accepted

## Date

2026-05-10

## Context

DocFormatter currently implements Stage 1 of the SciELO XML production pipeline: pre-formatting `.docx` files so the legacy SciELO Markup Word plugin can auto-tag them. Two problems motivated this feature:

1. **Stage 1 author extraction gaps**: For at least two articles (5313, 5449), Markup's `mark_authors` macro fails to auto-mark some or all authors. The failure is not in DocFormatter's `ExtractAuthorsRule` (the diagnostic for 5313 lists 11 authors with `confidence=high`); it happens downstream when Markup processes the `.docx` produced by Stage 1. The user has to manually re-mark authors, defeating the purpose of automatic markup.
2. **Stage 2 tags Markup does not handle well**: Several SciELO XML 4.0 tags either are not auto-marked by Markup or are marked incompletely (`elocation`, author `xref`/ORCID attributes, `corresp`, `abstract`, `kwdgrp`, `hist` with `received`/`accepted`/`pub`). These currently require manual intervention or fail downstream validation.

SciELO Markup has been unmaintained for ~3 years (per `docs/scielo_context/README.md`). This raises the strategic question: should DocFormatter expand to **help** Markup (pre-mark only what is safe) or **replace** Stage 2 of Markup entirely?

The decision space:
- **A — Incremental rollout helping Markup**: Add Stage 2 functionality that pre-marks tags Markup either ignores or handles incompletely, fixing the upstream Stage 1 author issue along the way. Respects existing anti-duplication invariants from `docs/scielo_context/REENTRANCE.md`.
- **B — Investigation-first**: Audit all 10 before/after example pairs exhaustively before writing code, then ship in one or two big releases.
- **C — Replace Stage 2**: Reimplement Markup's entire `.docx` → tagged-`.docx` pipeline in DocFormatter, eliminating the legacy dependency.

## Decision

Adopt approach **A — Incremental rollout helping Markup**, with releases ordered by implementation difficulty:

| Release  | Scope                                                                |
|----------|----------------------------------------------------------------------|
| Phase 1  | Fix Stage 1 author extraction so Markup's `mark_authors` succeeds on articles 5313 and 5449 (root-cause investigation + rule adjustment). |
| Phase 2  | Easy Stage 2 tags: `elocation` update, `[kwdgrp]`/`[kwd]`, `[abstract]` (`xmlabstr` with `language="en"`). |
| Phase 3  | Medium Stage 2 tags within the author block: `[corresp]`, author `xref ref-type="aff"`, `xref ref-type="corresp" rid="c1"`, `[authorid]` ORCID, attributes (`rid`, `corresp`, `deceased="n"`). Authors themselves remain Markup's responsibility (anti-duplication). |
| Phase 4  | Hard Stage 2 tag: `[hist]` with `received` / `accepted` / `histdate datetype="pub"`, ported from the date-parsing logic in [Gilcemir/Marcador_de_referencia](https://github.com/Gilcemir/Marcador_de_referencia/blob/master/BibliographyHandlers/AccessedOnHandler.cs). |

Each release is gated by byte-comparable diff success against the `examples/phase-2/{before,after}/` corpus (see ADR-003).

## Alternatives Considered

### Alternative B — Investigation-First Sprint

- **Description**: Run a dedicated investigation phase comparing all 10 before/after pairs across all candidate tags, produce an exhaustive specification, then implement in one or two large releases.
- **Pros**: Reduces uncertainty about total scope; surfaces edge cases early; potentially fewer architectural pivots mid-implementation.
- **Cons**: High upfront cost with no shipped value; delays the Stage 1 author fix that is the most acute pain today; encourages over-engineering ("must cover 100% of the diff before I can ship").
- **Why rejected**: The user's pain is concrete (5313/5449 author failures) and the corpus is small (10 pairs). A staged implementation already lets each release validate against the corpus, achieving the same de-risking benefit incrementally.

### Alternative C — Replace SciELO Markup Stage 2

- **Description**: Reimplement the entire `.docx` → tagged-`.docx` pipeline that Markup performs, including author tagging, keyword separation, footnote handling, etc. DocFormatter would emit a fully tagged document ready for `parser.exe`/`convert.exe`.
- **Pros**: Eliminates dependency on unmaintained legacy software; full control over output; future-proof.
- **Cons**: Markup's behavior lives in ~116k lines of legacy VBA. Replacing it requires reimplementing `mark_authors` separators (`;`/`,`/`&`/`and`/`y`/`e`/`et al`), the `markup_sup_as` superscript trap, normaff parsing, footnote linking, and more. Estimated 5–10× the effort of approach A. Forces a new round of validation against `parser.exe` directly.
- **Why rejected**: Out of scale for a solo project. The user's stated motivation is to reduce manual rework on specific tags, not to replace the toolchain. If Markup proves untenable in the future, approach A's modular rules become reusable building blocks for a Stage-2 replacement.

## Consequences

### Positive

- Each release ships independent value; failure of a later release does not invalidate earlier ones.
- Diff-gate against the `examples/phase-2/{before,after}/` corpus is automatable and gives objective release readiness signal.
- The Stage 1 author fix lands first, addressing the most acute pain.
- Respects existing decision documented in `docs/scielo_context/README.md` ("DocFormatter prepares Stage 2's input; it does not replace Stage 2").
- `docs/scielo_context/REENTRANCE.md` invariants (don't pre-mark `[author]`/`[fname]`/`[surname]`/`[normaff]`) remain valid. See follow-up note below for `[kwd]`.

## Follow-up note (2026-05-11): inner Phase-2 tags

The anti-duplication invariant originally listed `[kwd]` alongside `[author]`/`[fname]`/`[surname]`/`[normaff]`. Re-reading the `examples/phase-2/after/` corpus showed the AFTER shape carries `[sectitle]`, `[kwd]`, `[p]`, and `[email]` literals inside the Phase-2 wrappers, and Markup does not duplicate them when pre-marked. The narrowed invariant is:

- **Still owned by Markup (do NOT pre-mark)**: `[author]`, `[fname]`, `[surname]`, `[normaff]`.
- **Owned by Phase 2 (DO emit, in their own `Run`)**: `[sectitle]` and `[kwd]` inside `[kwdgrp]`, `[sectitle]` and `[p]` inside `[xmlabstr]`, `[email]` inside `[corresp]`.

Emitting each tag literal in its own OpenXML `Run` is required for the Word Markup VBA `color(tag)` mapping to paint per-tag colors; collapsing them into a single `Run` produces a monochromatic line. The Phase-2 emitters (`EmitKwdgrpTagRule`, `EmitAbstractTagRule`, `EmitCorrespTagRule`, `EmitAuthorXrefsRule`) now follow this pattern, mirroring `EmitHistTagRule`'s `RemoveAllChildren()` + per-segment `AppendChild()` approach.

### Negative

- DocFormatter remains coupled to Markup's quirks (must respect anti-duplication invariants in every new rule).
- If Markup is eventually deprecated upstream, this approach delays — but does not block — a future Stage-2 replacement.
- Four releases mean four validation cycles; cumulative overhead is higher than a single big-bang release.

### Risks

- **Risk**: A Phase-N release accidentally violates an anti-duplication invariant and breaks Phase-(N-1)'s diff.
  **Mitigation**: Diff gate runs across all 10 corpus pairs on every release, not just the new tags.
- **Risk**: The Phase 4 date-parsing port from Marcador_de_referencia is harder than expected and stalls the rollout.
  **Mitigation**: Phase 4 is intentionally last; failure to ship Phase 4 still leaves Phases 1–3 delivering value.

## Implementation Notes

- Pipeline invocation: a separate subcommand or flag (e.g., `docformatter phase2 input.docx`) per ADR-004 (to be drafted with TechSpec).
- Output directory: distinct from Phase 1's `examples/formatted/` (e.g., `examples/formatted-phase2/`) to keep diff comparisons clean.
- Failure policy: skip + warn, never abort (see ADR-002).
- Testing strategy and exact CLI shape are deferred to TechSpec.
