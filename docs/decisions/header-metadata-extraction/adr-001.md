# ADR-001: Esqueleto alinhado ao spec com 4 regras de extração

## Status

Accepted

## Date

2026-05-05

## Context

O DocFormatter (descrito em `instructions.md`) prevê um pipeline de 14 regras para normalizar artigos científicos `.docx` no formato editorial da revista. Implementar as 14 regras de uma vez exigiria validação custosa e adiaria entrega de valor. A equipe identificou que **4 campos** (Autores, DOI, Título, ELOCATION) concentram a maior parte do tempo manual atual — sendo Autores o mais doloroso, com regras complexas de split (`,` e ` and `), labels sobrescritos de afiliação e hyperlinks ORCID que precisam ser limpos antes do parse.

O escopo do MVP precisa equilibrar três forças:
1. **Velocidade de entrega**: critério de done é "1 artigo real funciona end-to-end", indicando apetite por entrega rápida.
2. **Risco de retrabalho**: regras adicionais (afiliações, abstract, keywords, body) virão em fases seguintes — uma arquitetura descartável forçaria reescrita.
3. **Feedback de qualidade**: parsing de autores tem incertezas inerentes; o sistema precisa sinalizar quando o resultado é duvidoso.

## Decision

Implementar o **esqueleto canônico do pipeline descrito no spec**, populado apenas com as **4 regras necessárias para os campos do MVP**, mais 1 regra custom de reescrita e 1 regra auxiliar para posicionamento de ELOCATION.

Componentes incluídos no MVP:
- `IFormattingRule`, `FormattingPipeline`, `FormattingContext`, `IReport`, `Report`, `RuleSeverity`, `FormattingOptions` — infraestrutura completa do pipeline.
- Regras de extração: `ExtractTopTableRule` (Critical), `ParseHeaderLinesRule` (Critical), `RemoveOrcidLinksRule` (Optional), `ParseAuthorsRule` (Optional).
- Regra custom: `RewriteHeaderMvpRule` (Critical) — reescreve apenas as 4 informações no topo, deixa o resto do documento intacto.
- Regra auxiliar: `LocateAbstractAndInsertElocationRule` (Optional) — localiza o parágrafo do Abstract e insere ELOCATION em uma linha imediatamente acima.
- CLI única com suporte a arquivo individual e pasta (batch).
- Relatório `.report.txt` sempre gerado; arquivo JSON de diagnóstico (`<nome>.diagnostic.json`) gerado **apenas** quando há `[WARN]` ou `[ERROR]` durante a execução.
- Build cross-platform: dev no macOS, deploy `.exe` self-contained `win-x64`.

## Alternatives Considered

### Alternative 1: Quick win sem arquitetura

- **Description**: CLI direta sem pipeline ou `FormattingContext`. Lógica de extração inline em um único arquivo, sem `Report` estruturado, sem JSON de diagnóstico.
- **Pros**: Entrega em ~1 semana. Mínimo de código.
- **Cons**: Adicionar a 5ª regra (afiliações) exige reescrever do zero. Sem `Report` formal, diagnóstico fica em `stdout`, perdido após a execução. Não respeita o desenho arquitetural do spec.
- **Why rejected**: O custo de jogar fora supera o ganho de uma semana, dado que o roadmap claramente caminha para mais regras. A escolha de `Best-effort + diagnóstico ruidoso` para autores exige `Report` formal.

### Alternative 2: Rede de segurança mais larga

- **Description**: Mesma base do esqueleto alinhado ao spec, mais: `DetectInputFormatRule` explícita (Critical), marcação visual no `.docx` via Word comments para autores incertos, golden file test em xUnit desde o dia 1, comparador semântico de `.docx`.
- **Pros**: Mais robustez antes de ir pra produção. Diagnóstico visual direto no documento (editor não precisa cruzar JSON com `.docx`).
- **Cons**: ~3-4 semanas vs 2-3. Atrasa o feedback de uso real, que é o método de validação escolhido pelo usuário ("1 artigo end-to-end").
- **Why rejected**: O critério de done escolhido é minimalista; o investimento em rede de segurança contradiz o apetite explícito por iteração rápida com casos reais. Pode ser revisitada na Fase 2 quando o volume de artigos processados aumentar.

## Consequences

### Positive

- Próximas regras (afiliações, abstract, keywords, body) entram no pipeline sem refactor.
- `FormattingContext` populado com 4 campos já fornece base para qualquer consumidor downstream futuro.
- `Report` + JSON de diagnóstico desde dia 1 cria hábito de logging estruturado.
- CLI com batch desde o MVP suporta o workflow de fornadas/edições.

### Negative

- Custo de ~1 semana extra vs Alternative 1.
- Sem `DetectInputFormatRule` explícita, a mensagem de erro quando a tabela superior está ausente vem de `ExtractTopTableRule` — pode ser menos clara que uma guarda dedicada.

### Risks

- **Risco**: parsing de autores produz resultado plausível mas errado em casos com nome contendo "and" ou ", Jr." → editor não percebe.
  - **Mitigação**: heurísticas conservadoras + diagnóstico JSON ruidoso quando o número de labels não bate ou quando há padrões ambíguos. Avaliar marcação visual no `.docx` (Word comments) na Fase 2 se erros silenciosos forem reportados.
- **Risco**: `LocateAbstractAndInsertElocationRule` não encontra o Abstract → ELOCATION não é escrita.
  - **Mitigação**: rule é Optional, gera `[WARN]` no report e diagnostic JSON; `.docx` sai sem ELOCATION mas com DOI/Título/Autores OK. Editor adiciona ELOCATION manual se preciso.

## Implementation Notes

- `FormattingOptions` carrega regex de DOI, separadores de autores (`,`, ` and `) e padrão de URL ORCID (`orcid.org`) hardcoded.
- Sem JSON de configuração externa — multi-perfil fica para fase posterior.
- `RewriteHeaderMvpRule` deve preservar superscript labels de afiliação ao escrever cada autor em sua própria linha.
- CLI distingue arquivo único de pasta automaticamente; em batch, gera `_batch_summary.txt`.
- Build target: `dotnet publish -r win-x64 -c Release --self-contained -p:PublishSingleFile=true -p:IncludeNativeLibrariesForSelfExtract=true -p:PublishReadyToRun=true -p:PublishTrimmed=false`.
