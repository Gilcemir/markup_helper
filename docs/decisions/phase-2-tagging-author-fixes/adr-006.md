# ADR-006: Diff Utility — Body-Text Extraction with Out-of-Scope Tag Stripping

## Status

Accepted

## Date

2026-05-10

## Context

ADR-003 establishes that each Phase 2 release passes when DocFormatter's output on each `examples/phase-2/before/<id>.docx` matches `examples/phase-2/after/<id>.docx` within the in-scope tags. The implementation question: how to actually compute that comparison?

Three implementation depths were considered:

- **A — Body-text extraction with out-of-scope stripping**: Extract a flat string from each document's body (every paragraph's runs concatenated, preserving SciELO tag literals like `[abstract …]…[/abstract]` exactly as they appear in text). Before comparing, remove every `[tag …]…[/tag]` pair from the `after` side that is NOT part of the current release's scope. Compare strings; on mismatch, report the first divergent position with surrounding context.
- **B — Structured parsing**: Build a small parser for the `[…]` delimiter syntax. Extract a tag tree from each side. Compare structurally (tag presence, attributes, child ordering). More accurate but more code.
- **C — Off-the-shelf diff library** (e.g., `DiffPlex`): Generate a Myers-diff or LCS-based output. Pretty, but does not solve the scoping problem.

The corpus is small (10 pairs), the tags are well-defined (DTD 4.0 vocabulary), and the gate is run pre-release rather than per-commit. Pragmatism beats sophistication.

## Decision

Implement approach **A — body-text extraction with out-of-scope tag stripping**.

The diff utility is exposed as `docformatter phase2-verify <before-dir> <after-dir>` (per ADR-005) and works as follows:

1. For each pair (matched by filename), run the full Phase 2 pipeline over `before/<id>.docx` to produce a `.docx` (in a temp directory).
2. Extract the body text from the produced `.docx` and from `after/<id>.docx` as flat strings, preserving SciELO tag literals exactly. Whitespace normalization: collapse repeated whitespace; trim per-paragraph; concatenate paragraphs with `\n`.
3. Strip from the `after/` text any `[tagname …]…[/tagname]` pair whose `tagname` is NOT in the **current release's in-scope tag set** (a static list updated per release in code or config). Do not strip from the produced text — it should already only contain in-scope tags.
4. String-compare. If equal: pass. If not: print the first divergent position (e.g., "differ at index 412") with ~80 chars of context on each side.

The release scope is encoded as a constant set per release (e.g., `Phase2Scope.Tags = ["elocation", "abstract", "kwdgrp"]` for the Phase 2 release). Each release that promotes adds its tags to the cumulative scope.

Output of `phase2-verify`:

```text
[PASS] 5136
[PASS] 5293
[FAIL] 5313
   diverge at offset 1247
   produced: ...end of abstract] [kwdgrp language="en"...
       after: ...end of abstract] [aff id="aff1"...
[PASS] 5419
...
Summary: 7/10 pass
```

A non-zero exit code is returned if any pair fails.

## Alternatives Considered

### Alternative B — Structured tag-tree parser

- **Description**: Lex the bracket syntax (`[`, `[/`, `]`), parse `tag attr="value"` opening tokens, build a tree, compare trees ignoring whitespace.
- **Pros**: More precise (e.g., handles attribute reordering, ignores immaterial whitespace differences). Out-of-scope filtering is natural at the tree level.
- **Cons**: Adds ~200-400 lines of parser + comparator code that has its own correctness risk. The DTD 4.0 syntax has edge cases (escaped quotes, etc.) that need handling. The corpus is too small to benefit from the precision uplift.
- **Why rejected**: Diminishing returns. If the body-text approach produces a false positive on a real corpus pair, the fix is usually a tighter whitespace normalization, not a full parser.

### Alternative C — `DiffPlex` library

- **Description**: Use NuGet `DiffPlex` for Myers/LCS diff with side-by-side rendering.
- **Pros**: Beautiful output; battle-tested.
- **Cons**: Does nothing about scoping (the core problem). Adds a dependency. Visual richness is wasted for an automated gate that mostly should pass and emit one short failure message.
- **Why rejected**: Solves the wrong problem. The diff is rarely viewed; pass/fail is the value.

## Consequences

### Positive

- Self-contained: no new dependencies; ~100 lines of code.
- Fast: 10 pairs × pipeline run is the dominant cost; diff itself is microseconds.
- Easy to evolve: if scope filtering becomes problematic, swap the strip step for a real parser without changing the call sites.

### Negative

- Whitespace differences (a real-world concern when OpenXML insertion creates extra spaces) can produce false positives.
  **Mitigation**: aggressive whitespace normalization step + manually inspect the first failure to decide if normalization needs tightening.
- Out-of-scope tag stripping is regex-based and could mis-strip a tag with unusual attributes.
  **Mitigation**: the SciELO bracket syntax is tightly constrained (no nested same-tag, no escaped delimiters); a careful regex covers it. Validate against the corpus pre-release.

### Risks

- **Risk**: The `after/` corpus contains tags the regex does not recognize as out-of-scope.
  **Mitigation**: When a previously-passing pair starts failing after a corpus update, inspect manually before changing code.
- **Risk**: The diff says "differ at offset N" but the offset is hard to map to a paragraph for the operator.
  **Mitigation**: include `~80 chars of context on each side` in the failure output. Sufficient for visual orientation.

## Implementation Notes

- Module location: `DocFormatter.Core/Reporting/Phase2DiffUtility.cs` (or similar — finalized in TechSpec).
- Body extraction: walk `MainDocumentPart.Document.Body.Elements<Paragraph>()` and concatenate `GetParagraphPlainText()` results (helper already exists per `RewriteAbstractRule.cs:443-460`).
- Whitespace normalization: trim per-paragraph; collapse `\s+` → single space; join with `\n`.
- Out-of-scope strip: regex `\[(\w+)([^\]]*)\](.*?)\[/\1\]` with non-greedy match + per-match check against in-scope set.
- In-scope set: `static readonly ImmutableHashSet<string> InScopeTags` per release scope class.
- Verification handler in CLI: `RunPhase2Verify(before, after)` → enumerates `*.docx` pairs, runs pipeline+diff for each, prints summary, sets exit code.
