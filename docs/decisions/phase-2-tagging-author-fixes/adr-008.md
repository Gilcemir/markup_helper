# ADR-008: Root Cause of Markup `mark_authors` Failure on 5313 / 5449

## Status

Accepted

## Date

2026-05-10

## Context

Articles `9_CR_5313_2.docx` (5313) and `1_AR_5449_2.docx` (5449) are extracted by DocFormatter Stage 1 with `confidence=high` for every author (11 authors and 2 authors respectively), yet when the operator runs the SciELO Markup `mark_authors` macro on the produced `.docx`, parts of the author block are not auto-tagged. The operator must re-mark each affected author manually, which defeats the purpose of Phase 1 ahead of Phase 2 work.

This ADR documents the root cause discovered during Phase 1 implementation, the fix decision, and the alternatives considered.

### Reproduction (per the task spec, subtask 1.1)

For both articles, the produced `examples/formatted/<id>.docx` re-shapes the original author paragraph into one paragraph per author. Each author paragraph contains:

1. A non-superscript run with the author name.
2. A superscript run with the comma-joined affiliation labels.
3. A non-superscript run with `" " + ORCID id`.

For 5313's affected author the produced superscript text is `1,*`. For 5449's affected author it is `1,2,*`. For an article that *does* pass Markup (5136), the equivalent author's produced superscript text is `1*` — no comma between the digit(s) and the corresp asterisk.

### Diff at the OpenXML level (subtask 1.2)

5136 (passes Markup) — original author paragraph for the affected author:

```
<w:r><w:rPr><w:vertAlign w:val="superscript"/></w:rPr><w:t>1*</w:t></w:r>
```

The digit and the asterisk live in a single superscript run. `ExtractAuthorsRule.SplitLabels` returns `["1*"]`, and `RewriteHeaderMvpRule.BuildAuthorParagraph` emits a single superscript run with text `1*`. Markup-friendly.

5313 (fails Markup) — original author paragraph for `Flavia Alves Silva`:

```
<w:r><w:rPr><w:vertAlign w:val="superscript"/></w:rPr><w:t>1</w:t></w:r>
<w:r><w:rPr><w:vertAlign w:val="superscript"/></w:rPr><w:t>*</w:t></w:r>
```

Two adjacent superscript runs. `ExtractAuthorsRule.AppendRunTokens` emits two `Label` tokens (`"1"`, `"*"`). The author's `AffiliationLabels` becomes `["1", "*"]`. `RewriteHeaderMvpRule.BuildAuthorParagraph` joins the labels with `","`, producing the superscript text `1,*`.

5449 (fails Markup) — same shape with `1,2` in the first run:

```
<w:r><w:rPr><w:vertAlign w:val="superscript"/></w:rPr><w:t>1,2</w:t></w:r>
<w:r><w:rPr><w:vertAlign w:val="superscript"/></w:rPr><w:t>*</w:t></w:r>
```

`SplitLabels("1,2")` returns `["1", "2"]`. The asterisk run produces `["*"]`. `AffiliationLabels` becomes `["1", "2", "*"]`. The joined superscript becomes `1,2,*`.

### Why Markup mis-handles the comma form

`mark_authors` (`PC-Programs/_analysis/markup_macros.txt:4799`) normalises the author range text by replacing `" and "`, `" & "`, `" y "`, `" e "`, `" &amp; "` with a marker, then chooses an author separator from `,`/`;`/space. When the input contains `,` and no `;`, it sets `sep_authors = ","` and `Split(authorstext, sep_authors)`. The comma inside the produced `1,*` (or `1,2,*`) label is therefore treated as an inter-author boundary. The asterisk segment becomes its own would-be "author" entry, fails `markup_surname_and_fname`, and the surrounding authors fail to mark cleanly.

### Wider impact than originally scoped

Investigating all ten Phase 2 corpus articles plus 5548 revealed that **eight** files in `examples/formatted/` currently produce a `1,*` (or `1,2,*`) superscript: `1_AR_5449_2.docx` (5449), `9_CR_5313_2.docx` (5313), and additionally `2_AR_5424_3.docx`, `3_AR_5293_3.docx`, `4_AR_5419_4.docx`, `5_AR_5434_3.docx`, `8_AR_5458_1.docx`, `10_AR_5549_2.docx`. Two articles (`7_CR_5136_3.docx`, `11_AR_5548_1.docx`) already produce the canonical `1*` shape because the original author paragraph stored the digit and the asterisk in a single superscript run. The user has only manually re-tested 5313 and 5449 with Markup, but the same root-cause bug presumably affects every article in the `1,*` set.

## Decision

Modify `ExtractAuthorsRule.AuthorBuilder.AddLabel` so that any label consisting solely of corresp-marker characters (`*`) is **appended onto the trailing characters of the previous label** rather than added as a new entry to `AffiliationLabels`. This is the single locus where adjacent superscript runs are folded into the rule's per-author label list.

After the fix:

- Input `[SUP]'1'` then `[SUP]'*'` produces `AffiliationLabels = ["1*"]`.
- Input `[SUP]'1,2'` then `[SUP]'*'` produces `AffiliationLabels = ["1", "2*"]`.
- Input `[SUP]'1*'` (single run) is unchanged; `SplitLabels` returns `["1*"]`, no merge needed.
- Input `[SUP]'*'` with no preceding numeric label keeps `["*"]`; merging requires a previous label.
- Input `[SUP]'1,*'` (single comma-separated run) is split by `SplitLabels` into `["1", "*"]`, then merged to `["1*"]`.

`RewriteHeaderMvpRule.BuildAuthorParagraph` then comma-joins `["1", "2*"]` to `1,2*`, identical to the canonical 5136 shape that Markup auto-marks correctly.

The fix is contained inside `ExtractAuthorsRule.cs`. `RewriteHeaderMvpRule` and the diagnostic writer require no behavioral change; the diagnostic JSON for affected articles will reflect the merged label form (e.g. `["1*"]` instead of `["1", "*"]`), which is consistent with how 5136 already reports.

## Alternatives Considered

### Alternative A — Change the comma-join in `RewriteHeaderMvpRule`

- **Description**: Leave `AffiliationLabels` alone. In `BuildAuthorParagraph`, when emitting the superscript label, special-case purely-asterisk labels and append them to the previous label string without a comma. So `["1", "2", "*"]` → emitted text `1,2*`.
- **Pros**: Smaller change; data model untouched; diagnostic JSON for non-5313/5449 articles unchanged.
- **Cons**: The task specification scopes the fix to `ExtractAuthorsRule.cs` or `HeaderParagraphLocator.cs` — `RewriteHeaderMvpRule.cs` is in "Dependent Files" (verify-only), not "Relevant Files". This alternative violates the scoping rule.
- **Why rejected**: Scope conflict with the task spec. Also, fixing the data model is more honest: the asterisk *is* a corresp marker attached to the preceding affiliation label, not a separate entry in the affiliation list.

### Alternative B — Add a separate `CorrespMarker` field on `Author`

- **Description**: Detect the asterisk in `ExtractAuthorsRule` and store it on a new `Author.CorrespMarker` property. Drop the asterisk from `AffiliationLabels` entirely.
- **Pros**: Cleanest semantic separation. Phase 3's xref/corresp rule consumes `CorrespMarker` directly.
- **Cons**: Larger change; touches `Author.cs`, `DiagnosticAuthor`, every rule that reads affiliation labels, and existing tests. Breaks the task spec's "MUST keep the rule's confidence semantics intact" + "same author count and label set" for non-affected articles, since the label set changes for *every* article whose author has an asterisk (currently 8 / 11 articles in the corpus).
- **Why rejected**: Beyond Phase 1 scope. Phase 3 (task_07) can split a merged label like `"2*"` into its aff vs corresp components when implementing the xref rule; that is a Phase 3 concern, not Phase 1's.

### Alternative C — Change Markup's `mark_authors` macro

- **Description**: Patch `markup_macros.txt:4799` so the comma-split logic ignores commas inside superscript label runs.
- **Pros**: Fixes the root cause at the layer that owns the bug.
- **Cons**: SciELO Markup is unmaintained for ~3 years (per `docs/scielo_context/README.md`); the project has explicitly chosen to *help* Markup, not modify it (ADR-001).
- **Why rejected**: Out of scope; violates ADR-001.

## Consequences

### Positive

- Markup auto-marks every author on the post-fix `.docx` for 5313 and 5449.
- The same fix incidentally improves Markup compatibility for the six other corpus articles whose author paragraph carries an adjacent corresp asterisk superscript run (5293, 5419, 5424, 5434, 5458, 5549). Since those articles share the same broken shape, they would have failed Markup the same way had the operator tested them.
- The post-fix label representation matches what 5136 already produces, eliminating an inconsistency in the corpus.
- Confidence semantics are intact: every affected author keeps `confidence=high`; only the label string-representation changes (e.g. `["1", "*"]` → `["1*"]`).

### Negative

- The `AffiliationLabels` set for 8 of 11 corpus articles changes shape (`["…", "*"]` → `["…*"]`). The pre-fix `examples/formatted/*.diagnostic.json` snapshot must be regenerated. The change is consistent with the canonical 5136 shape and lossless (the asterisk is preserved, just merged with the trailing affiliation label).
- Phase 3 (`EmitAuthorXrefsRule`, task_07) must split a merged label like `"2*"` into the affiliation component (`"2"`) and the corresp marker (`"*"`) when computing per-author `xref` and `rid` attributes. This is straightforward — a single `TrimEnd('*')` plus a "had asterisk" flag — and is captured as a follow-up note in the task memory.

### Risks

- **Risk**: A future Phase 2 article carries a non-asterisk corresp marker (e.g. `†`, `‡`, `§`). The fix recognises only `*` and would not merge those.
  **Mitigation**: extend `IsCorrespMarker` to whitelist additional characters when such a marker first appears in the corpus. Document any addition with a follow-up ADR.
- **Risk**: The fix accidentally merges a label that *should* remain separate (e.g. an aff label that legitimately starts with `*`).
  **Mitigation**: the merge predicate requires the entire trimmed label to consist of asterisk character(s). Numeric labels and mixed labels are untouched. Added unit tests cover the boundary.

## Implementation Notes

- Single change-site: `AuthorBuilder.AddLabel` in `DocFormatter.Core/Rules/ExtractAuthorsRule.cs`.
- Recognition predicate: a label is a corresp marker when, after trimming whitespace, every remaining character is `*`. Empty labels are dropped (no-op) since `SplitLabels` already filters them.
- Merge action: `Labels[^1] = Labels[^1] + trimmed` only when the previous label exists. If the asterisk-only label is the first label collected, it is kept as-is to preserve information for Phase 3.
- No new public types or methods. No change to `Author` record, `DiagnosticAuthor`, `RewriteHeaderMvpRule`, or `DiagnosticWriter`.

## Manual Markup verification (subtask 1.7)

The operator must open the post-fix `examples/formatted/9_CR_5313_2.docx` and `examples/formatted/1_AR_5449_2.docx` in Word with `markup.prg` loaded, run `*authors`, and confirm every author auto-marks without manual re-marking. Result to be appended to this ADR (or release notes) once verified.

> **Verification status (2026-05-10)**: Pending manual operator verification. The fix has been validated at the unit-test level (regression fixtures for both shapes pass) and at the corpus-snapshot level (post-fix diagnostic JSON for 5313/5449/+6 changes only the `affiliationLabels` shape; author count, names, ORCIDs, and confidence are unchanged across all 11 corpus articles).
