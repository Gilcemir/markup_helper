# ADR-004: Pipeline Organization — Reuse `FormattingPipeline` with DI-Selected Rule Sets

## Status

Accepted

## Date

2026-05-10

## Context

Phase 2 introduces a new class of work — emitting SciELO tag literals (`[tag attr="v"]…[/tag]`) into the `.docx`, which no current rule does. The natural design question: should Phase 2 reuse the existing pipeline infrastructure or run its own?

The current pipeline (in `DocFormatter.Core/Pipeline/`) consists of three stable abstractions:
- `IFormattingRule` (file: `IFormattingRule.cs:5-12`): `void Apply(WordprocessingDocument, FormattingContext, IReport)`.
- `FormattingPipeline` (file: `FormattingPipeline.cs`): orchestrates an `IEnumerable<IFormattingRule>` and routes per-rule failures based on `RuleSeverity`.
- `FormattingContext` (file: `FormattingContext.cs`): mutable shared state across a single document run.

Composition happens in `DocFormatter.Cli/CliApp.cs:199-221` via `Microsoft.Extensions.DependencyInjection.IServiceCollection`. Each rule is `services.AddTransient<IFormattingRule, RuleClass>()`.

`FormattingPipeline` is intrinsically rule-agnostic — it does nothing Phase-1-specific. The orchestration code, the report wiring, the diagnostic writing, the OpenXML lifecycle, all generalize to any sequence of `IFormattingRule` implementations.

## Decision

Phase 2 reuses `FormattingPipeline`, `IFormattingRule`, `FormattingContext`, `IReport`, and `DiagnosticWriter` as-is. Phase 2 rules are new `IFormattingRule` implementations placed in `DocFormatter.Core/Rules/Phase2/`. Rule-set selection happens at DI-composition time via dedicated extension methods on `IServiceCollection`:

```csharp
// New methods (illustrative, exact shape decided in TechSpec)
public static IServiceCollection AddPhase1Rules(this IServiceCollection s) { /* current 11 rules */ }
public static IServiceCollection AddPhase2Rules(this IServiceCollection s) { /* new Phase 2 rules */ }
```

The CLI dispatcher (per ADR-005) calls one or the other depending on the subcommand. The same `FormattingPipeline` consumes whichever set was registered.

`FormattingContext` is extended cautiously: only when a Phase 2 rule needs structured data that crosses rule boundaries does it gain a new field. Rules that need internal-only state keep it inside themselves. New structured types (Affiliation, Corresp, Hist, Keywords, Abstract metadata) live in `DocFormatter.Core/Models/Phase2/` (or similar) and are referenced by the context only when shared.

## Alternatives Considered

### Alternative A — Parallel `Phase2Pipeline` with its own `IPhase2Rule` and `Phase2Context`

- **Description**: Build a completely separate orchestrator. New interface `IPhase2Rule`, new context, new pipeline class, parallel DI registration.
- **Pros**: Strong isolation. Phase 1 changes cannot break Phase 2 and vice versa. Phase-2-specific concerns (tag emission helpers, in-scope tag tracking) live entirely in Phase 2 territory.
- **Cons**: Duplicates `FormattingPipeline` orchestration code (severity routing, exception capture). Duplicates report/diagnostic wiring. Doubles the maintenance surface for a feature that conceptually does the same thing as Phase 1: walk a docx, mutate it, report.
- **Why rejected**: The duplication has no architectural payoff. The two pipelines do the same kind of work; differences (tag emission, before/after diff) are rule-level concerns, not orchestrator concerns.

### Alternative B — Reuse `FormattingPipeline` + bloat `FormattingContext` with optional Phase 2 fields

- **Description**: Mix Phase 1 and Phase 2 rules in the same pipeline. Add `Affiliations?`, `Hist?`, `Keywords?` directly to `FormattingContext`; rules check for non-null before using.
- **Pros**: Trivially simple wiring; no DI extension methods needed.
- **Cons**: `FormattingContext` becomes the union of every concern across phases — high entropy, low cohesion. Phase-2 fields surface in Phase-1 diagnostics by accident (or at least look broken when null). No way to run Phase 2 rules without also running Phase 1 (or vice versa) without per-rule guards.
- **Why rejected**: Pollutes a stable type for short-term convenience. Makes "run only Phase 2 on a Phase 1 output" awkward to express.

## Consequences

### Positive

- 100% reuse of existing orchestration; only new rule code is new code.
- DI-based selection lets each subcommand express precisely which rules it wants (e.g., `phase2-verify` could reuse the same Phase 2 rule set).
- Phase 2 rules unit-test the same way Phase 1 rules do today (`AuthorsParagraphFactory`/`Phase2DocxFixtureBuilder` patterns).

### Negative

- Shared `FormattingContext`: any field added for Phase 2 is visible to Phase 1 rules and vice versa. Discipline required.
- Composition-time selection means any caller (test or CLI) needs to pick a rule set explicitly; no single "default everything" path.

### Risks

- **Risk**: A Phase 2 rule accidentally relies on a Phase 1 context field that was set only because Phase 1 ran first.
  **Mitigation**: Document on each Phase 2 rule which `FormattingContext` fields it expects to be populated and whether it tolerates missing ones. The skip-and-warn policy (ADR-002) handles missing inputs.

## Implementation Notes

- New folder: `DocFormatter.Core/Rules/Phase2/`. Existing rules stay where they are.
- Extension methods in `DocFormatter.Cli/CliApp.cs` (or a new `DependencyInjection.cs` file in the Cli or Core project) for `AddPhase1Rules` / `AddPhase2Rules`.
- New shared models, when needed, in `DocFormatter.Core/Models/Phase2/`. Each rule owns its private state internally.
- Naming convention for Phase 2 rules: `EmitXxxTagRule` (e.g., `EmitElocationTagRule`, `EmitAbstractTagRule`).
