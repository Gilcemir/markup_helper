# ADR-003: ORCID extraction overrides spec's "remove" behavior

## Status

Accepted

## Date

2026-05-05

## Context

The original DocFormatter spec (`instructions.md`) defines `RemoveOrcidLinksRule` with the behavior: *"Hiperlinks ORCID (URL contém orcid.org): removidos por completo — texto, ícone e relacionamento."* The first PRD draft inherited this behavior, treating ORCID as something to strip.

During TechSpec grilling the user clarified the actual editorial intent: the ORCID identifier carries value as a permanent author identifier and must be **preserved as plain text** alongside the author name. Sample expected output: `José Silva¹ 0000-0002-1825-0097`. The hyperlink itself, however, must be removed (so the output is plain text, not a clickable link to orcid.org), as must any orphan ORCID badge image. When no ORCID is present, the author renders as `José Silva¹` unchanged.

This is a behavioral divergence from the original spec significant enough to warrant an explicit decision record.

## Decision

The MVP implements **ORCID extraction**, not removal:

1. Walk the authors paragraph looking for `<w:hyperlink>` elements.
2. For each hyperlink, resolve `r:id` against the document's relationship part to obtain the target URL.
3. If the URL contains `orcid.org`, apply regex `\d{4}-\d{4}-\d{4}-\d{3}[\dX]` to extract the ORCID identifier.
4. Replace the entire `<w:hyperlink>` element with a plain `<w:r><w:t>` containing the extracted ID. Remove the corresponding relationship from `document.xml.rels`.
5. Detect and remove any orphan inline image (`<w:drawing>`) that immediately follows the original hyperlink position and whose alt text or relationship target points to ORCID assets (commonly the green ORCID badge).
6. Attach the extracted ID to the matching author record produced downstream by `ParseAuthorsRule`.

The rule is renamed from `RemoveOrcidLinksRule` (spec) to `ExtractOrcidLinksRule` to reflect the new intent. Severity remains `Optional`: failure to extract or clean produces `[WARN]` entries but does not abort the pipeline.

The PRD's Core Features section #3, the Technical Constraints section, and the rule's purpose have been updated to reflect this decision.

## Alternatives Considered

### Alternative 1: Remove ORCID completely (spec default)

- **Description**: Strip the ORCID hyperlink, text, relationship, and icon — match the spec verbatim.
- **Pros**: Zero divergence from the original spec; simpler implementation (no regex extraction, no per-author attachment).
- **Cons**: Loses an identifier the editorial workflow values. Users would have to re-enter ORCID IDs manually downstream — defeating the MVP's goal of cutting manual time.
- **Why rejected**: User explicitly asked for the ID to be preserved.

### Alternative 2: Keep hyperlink, just guard against breakage

- **Description**: Leave the `<w:hyperlink>` intact, only remove the orphan icon. ORCID remains clickable.
- **Pros**: Minimal mutation of the document structure.
- **Cons**: The downstream editorial flow expects plain text per the user's spec ("0000-0002-1825-0097" after the name). Hyperlinks complicate later automated processing and inconsistent with how DOI is rendered (plain text).
- **Why rejected**: Violates the user's expected output format.

## Consequences

### Positive

- Editorial flow preserves the ORCID identifier without manual re-entry.
- Output is consistent: DOI, ELOCATION, and ORCID all render as plain text.
- Non-ORCID hyperlinks (e.g., links the author manually inserted) remain untouched, matching the spec's other constraint.

### Negative

- Implementation is meaningfully more complex than pure removal (~20 extra lines: relationship lookup + regex extraction + per-author attachment + orphan icon detection).
- A regex change in ORCID format (none expected — the format has been stable since 2012) would require a code change.

### Risks

- **Risk**: Hyperlink target URL is the file path (e.g., `file:///C:/.../article.docx#orcid`) instead of `https://orcid.org/...`. The user explicitly mentioned this case ("um link do próprio arquivo").
  - **Mitigation**: Detection is by URL substring `orcid.org`, which catches both `https://orcid.org/...` and pseudo-targets that include the substring. If the URL does not contain `orcid.org`, the link is not treated as ORCID and goes through generic hyperlink handling (out of scope for MVP — the link stays in the document).
- **Risk**: Multiple ORCID-like patterns in a single hyperlink target (rare).
  - **Mitigation**: Take the first regex match. Log `[WARN]` if more than one pattern is found.
- **Risk**: ORCID ID without hyperlink (typed directly in the document).
  - **Mitigation**: Out of scope for MVP. The rule only acts on `<w:hyperlink>` elements. Plain-text ORCID IDs in author lines pass through as part of the name.

## Implementation Notes

- Regex: `\b\d{4}-\d{4}-\d{4}-\d{3}[\dX]\b` (case-insensitive on the trailing X).
- The rule writes to `Author.OrcidId` on the author record. `RewriteHeaderMvpRule` reads this field and emits the plain-text ID after the affiliation labels.
- Orphan icon detection: any `<w:drawing>` whose embedded blip references the same `r:id` as the removed hyperlink (Word commonly nests the badge inside the hyperlink) is removed when the hyperlink is converted; freestanding badges immediately preceding/following the removed hyperlink are not touched in the MVP.
