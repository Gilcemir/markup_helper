# ADR-005: CLI Dispatch ŌĆö Hand-Rolled Subcommands `phase2` and `phase2-verify`

## Status

Accepted

## Date

2026-05-10

## Context

The current CLI (`DocFormatter.Cli/CliApp.cs:21-75`) is hand-rolled. The first positional argument is interpreted: either a recognized flag (`-h`, `--help`, `--version`) or a path. There is no subcommand layer, no flag parser library, no help generator beyond a bespoke usage text.

Phase 2 introduces at least two new operations that need to be reachable from the CLI:

1. Run the Phase 2 pipeline: `ŌĆ” <input.docx>` ŌåÆ emits a `.docx` with Phase 2 tags pre-marked.
2. Run the Phase 2 diff gate: `ŌĆ” <before-dir> <after-dir>` (or similar) ŌåÆ reports how many corpus pairs match.

Phase 1 already has a default invocation (`docformatter <path>`) that the user relies on. Backward compatibility is non-negotiable.

Three approaches were considered:

- **A ŌĆö Hand-rolled subcommand layer**: extend `CliApp.cs` with a first-token dispatcher. `phase2` and `phase2-verify` become recognized first tokens.
- **B ŌĆö Global flag**: add `--stage=2` (or `--phase2`). Default = Phase 1.
- **C ŌĆö Adopt `System.CommandLine`** (or Cocona): rewrite the parser using a library that gives subcommands, options, validation, and `--help` for free.

## Decision

Adopt approach **A ŌĆö hand-rolled subcommand dispatch**. The `CliApp.Run()` entry point inspects the first non-flag argument: if it matches a known subcommand (`phase2`, `phase2-verify`), it routes to a corresponding handler. Otherwise the existing Phase 1 handling applies.

Backward compatibility: `docformatter file.docx`, `docformatter directory/`, `docformatter -h`, `docformatter --version` continue to behave exactly as before.

New surface (illustrative, finalized in TechSpec):

| Command                                            | Behavior                                                                                                |
|----------------------------------------------------|---------------------------------------------------------------------------------------------------------|
| `docformatter <input>`                             | Phase 1 (existing). Output ŌåÆ `<sourceDir>/formatted/`.                                                  |
| `docformatter phase2 <input>`                      | Phase 2 pipeline. Output ŌåÆ `<sourceDir>/formatted-phase2/`.                                             |
| `docformatter phase2-verify <before> <after>`      | Run Phase 2 over each `<before>/*.docx`; diff against `<after>/*.docx`; print pass/fail + first divergence. |

`<input>` follows the existing rules: a single `.docx` file path or a directory containing `.docx` files (handled via the same `RunSingleFile` / `RunBatch` split as Phase 1).

## Alternatives Considered

### Alternative B ŌĆö Global flag `--stage=2` / `--phase2`

- **Description**: One CLI verb, modal flag. `docformatter --stage=2 input.docx`.
- **Pros**: Smallest delta from current parsing.
- **Cons**: Mixes mode-of-pipeline with a positional argument. Verbose to extend (e.g., `phase2-verify` would need yet another flag like `--verify-against=<dir>`). Reads less naturally than `phase2` as a subcommand. Most CLIs grow subcommands eventually; pre-empting that growth is cheap now.
- **Why rejected**: Subcommands scale better as more Phase 2 operations appear (e.g., a future `phase2-stats`).

### Alternative C ŌĆö Adopt `System.CommandLine`

- **Description**: Replace the hand-rolled parser. Get subcommands, validation, help, suggestions for free.
- **Pros**: Cleaner code; built-in `--help` per subcommand; idiomatic for new .NET CLIs.
- **Cons**: `System.CommandLine` is large and the API is still in flux at certain versions. Adopting it requires migrating the existing Phase 1 invocation surface, including `RunSingleFile` / `RunBatch` arg handling. That work is unrelated to Phase 2's value and would block the MVP. The existing hand-rolled code is small (`CliApp.cs:21-75`); the marginal benefit of a library is not yet earned.
- **Why rejected**: Scope creep. If migration to `System.CommandLine` is desired later, that becomes its own ADR.

## Consequences

### Positive

- Backward-compatible with all existing invocations.
- Adding more Phase 2 operations is straightforward (new branch in the dispatcher).
- Zero new dependencies.

### Negative

- The hand-rolled dispatcher must be maintained alongside the existing arg handling. If the subcommand list grows past ~5, refactoring to a library becomes attractive.
- No automatic `--help <subcommand>` rendering. Each subcommand handler must print its own usage when invoked with `-h`.

### Risks

- **Risk**: The dispatcher accidentally swallows a path argument that happens to equal `phase2` or `phase2-verify`.
  **Mitigation**: Validate at dispatch time that no `.docx` extension or directory exists matching that token before treating it as a subcommand. (Documenting this nuance as a known sharp edge of hand-rolled parsing.)
- **Risk**: Help output diverges across subcommands as new ones are added.
  **Mitigation**: Centralize usage strings as constants in a single file.

## Implementation Notes

- Add subcommand names as constants (e.g., `static class Subcommands { public const string Phase2 = "phase2"; ŌĆ” }`).
- The dispatcher is at the top of `CliApp.Run()`, before the existing first-token logic.
- Each subcommand has its own handler method (`RunPhase2`, `RunPhase2Verify`) that mirrors `RunSingleFile`/`RunBatch` patterns.
- Help text per subcommand is printed when the subcommand is invoked with `-h` or with no further arguments.
