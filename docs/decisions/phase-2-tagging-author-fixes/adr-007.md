# ADR-007: Phase 4 Date-Parser Port — Rewrite from Scratch Using `Marcador_de_referencia` as Reference

## Status

Accepted

## Date

2026-05-10

## Context

Phase 4 (per ADR-001) emits the `[hist]` block: `received` (required), `revised*`, `accepted?`, plus `[histdate datetype="pub"]`. Each child carries `dateiso="YYYYMMDD"` zero-padded when month or day is missing. To populate these dates, the rule must parse natural-language date phrases such as "Received on 12 March 2024", "Accepted: 2024-04-15", or "Published 2024".

The user maintains a separate C# project, `Marcador_de_referencia`, that already implements analogous date detection in another context (bibliography references):

> https://github.com/Gilcemir/Marcador_de_referencia/blob/master/BibliographyHandlers/AccessedOnHandler.cs

Three integration paths are possible:

1. **Rewrite**: implement from scratch in `DocFormatter.Core`, using the original code as algorithmic reference only.
2. **Copy**: bring the relevant file(s) into `DocFormatter.Core/Rules/Phase2/HistDateParsing/` with a header citing origin.
3. **Vendor as Git submodule**: pull `Marcador_de_referencia` as a subtree.
4. **Defer**: postpone the decision until Phase 4 starts.

`DocFormatter` enforces `Nullable=enable`, `TreatWarningsAsErrors=true`, Serilog-based logging, and the Microsoft.Extensions.DI composition pattern. `Marcador_de_referencia` was written under different assumptions (different framework target, different logging, different conventions), evidenced by the user's confirmation during clarification.

## Decision

For Phase 4, **rewrite the date-parsing logic from scratch** in `DocFormatter.Core/Rules/Phase2/HistDateParsing/` (or similar). Use `BibliographyHandlers/AccessedOnHandler.cs` as algorithmic reference (which date phrases to recognize, the regex shape, the canonicalization to ISO 8601). Do NOT copy code verbatim or vendor the repo.

Concretely:

- Read `AccessedOnHandler.cs` to extract the inventory of date phrases recognized and the canonicalization rules.
- Implement a `HistDateParser` (single class, public static methods) inside `DocFormatter.Core` that consumes plain text or a `Paragraph`, returns either `(DateOnly date, string raw, ConfidenceLevel confidence)` or null when no match.
- Format `dateiso` as required by SciELO: `YYYYMMDD`, with `00` substituted for missing month or day (per `docs/scielo_context/README.md` invariant).
- The Phase 4 emitter rule (`EmitHistTagRule`) calls `HistDateParser` per candidate paragraph and assembles the `[hist]` block.

The decision applies at Phase 4 kickoff. Phases 1–3 do not depend on this.

## Alternatives Considered

### Alternative — Direct file copy

- **Description**: Copy `AccessedOnHandler.cs` (and any helpers) into `DocFormatter.Core/Rules/Phase2/HistDateParsing/`, add a header comment citing the origin commit hash.
- **Pros**: Fastest start; logic is already proven in production for the user.
- **Cons**: Ports along assumptions about framework target, namespaces, logging hooks, nullability annotations, and exception model. Likely 50%+ of lines need adaptation. The original repo's history and license metadata blur. Future maintenance has two homes.
- **Why rejected**: The savings are smaller than they look once adaptation overhead is accounted for. Rewriting with the original as a reference produces idiomatic DocFormatter code without dragging foreign conventions in.

### Alternative — Git submodule / subtree

- **Description**: Add `Marcador_de_referencia` as a submodule and reference its types from `DocFormatter.Core`.
- **Pros**: Single source of truth across both projects.
- **Cons**: Both projects evolve at different cadences and serve different use cases. Coupling them creates a release dependency. The relevant code is one file (~few hundred lines at most).
- **Why rejected**: Overkill. The two projects have no shared business reason to live together.

### Alternative — Defer to Phase 4 kickoff

- **Description**: Capture as Open Question in the TechSpec; decide when Phase 4 begins.
- **Pros**: Reflects honest uncertainty.
- **Cons**: The user has already signaled the preferred path during clarification. Capturing it now lets the rest of the TechSpec reference a known plan; deferring leaves a hole.
- **Why rejected**: The decision is made; documenting it now is cheap insurance.

## Consequences

### Positive

- Phase 4 code matches DocFormatter conventions from line 1.
- No dependency on `Marcador_de_referencia`'s release cadence.
- Reviewing and testing is constrained to one repository.
- Forces a fresh inventory of supported date phrases — possibly catching cases the original did not handle but Phase 4 needs.

### Negative

- Modest duplication of effort: the user wrote the algorithm once, will write it again (in better-fit code).
- Risk of forgetting an edge case the original handled. Mitigation: maintain a unit-test inventory derived from `AccessedOnHandler.cs` + corpus samples.

### Risks

- **Risk**: The rewrite ends up subtly different from the original and produces different ISO dates on edge cases.
  **Mitigation**: write unit tests covering every phrase shape from the reference file before implementing; the tests serve as a behavior contract.
- **Risk**: A pattern in the SciELO corpus is not recognized by either the original or the rewrite, producing a Phase 4 skip-and-warn miss.
  **Mitigation**: per ADR-002, missed dates result in `[hist]` being skipped, not in invalid markup. Operator handles manually; the next iteration adds the pattern to the parser.

## Implementation Notes

- Create unit tests first from a phrase inventory (TDD-friendly).
- `HistDateParser` is internal to `DocFormatter.Core/Rules/Phase2/HistDateParsing/`. The Phase 4 emitter rule (`EmitHistTagRule`) is its only caller.
- Date format conversion to `YYYYMMDD` with zero-pad lives inside `HistDateParser`, not in the rule. The rule receives ISO-ready strings.
- Logging via Serilog (consistent with the rest of `DocFormatter.Core`).
- No new NuGet packages.
