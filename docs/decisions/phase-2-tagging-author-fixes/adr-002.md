# ADR-002: Failure Policy for Phase 2 Rules — Skip and Warn

## Status

Accepted

## Date

2026-05-10

## Context

Every Stage 2 rule (and the Stage 1 author fix) operates on heuristics: locate a section by heading text, detect a date pattern, identify a keyword separator, find an ORCID hyperlink. Heuristics fail. The question is what a rule should do when it cannot identify with confidence what to mark.

Three behaviors are possible:

1. **Skip and warn**: Do not insert the tag. Record the miss in `diagnostic.json`. Pipeline continues. Markup or the human operator handles it downstream.
2. **Partial markup**: Insert what was confidently identified, leave the rest. For example, emit `[hist][received].../[received][/hist]` even when `accepted` was not detected.
3. **Abort**: Treat the miss as a critical error; halt the pipeline.

The SciELO Markup plugin auto-marks several tags (`[author]`, `[fname]`, `[surname]`, `[kwd]`, `[normaff]`) without checking for prior markup. Pre-marking incorrectly produces duplicate tags downstream — strictly worse than not marking at all (per `docs/scielo_context/REENTRANCE.md`). DocFormatter's job is to **reduce** manual work, not to replace human judgment.

## Decision

When a Phase 2 rule cannot identify its target with high confidence, it MUST skip insertion, log a structured warning to `diagnostic.json`, and let the pipeline continue. The rule MUST NOT emit partial or speculative markup.

The warning entry MUST include at minimum:
- The rule name
- A machine-readable reason code (e.g., `abstract_heading_not_found`, `hist_dates_unrecognized`)
- A human-readable message explaining what was missing

`diagnostic.json` already has an `issues` array; this policy extends its existing usage.

## Alternatives Considered

### Alternative A — Partial markup with confidence labels

- **Description**: Each rule emits whatever fragment it identified confidently and labels it (e.g., `confidence="medium"`).
- **Pros**: Preserves partial value; reduces operator's manual work in cases where the rule got most of the data right.
- **Cons**: SciELO DTD 4.0 has strict tag-content requirements (e.g., `[hist]` must contain `received`, with optional `revised*` and `accepted?` in that exact order). A partial `[hist]` missing required children produces invalid output. Adds a per-rule judgment of "what counts as partial-but-valid" that compounds maintenance.
- **Why rejected**: The downstream `parser.exe` validates against DTD; emitting partial-but-malformed tags trades one class of manual work for a worse one (debugging validation failures). Several pre-marked tags also risk duplication if Markup auto-marks the rest.

### Alternative B — Abort on first failure

- **Description**: A failed rule produces a `Critical` severity issue and halts the pipeline.
- **Pros**: Loud and obvious; impossible to silently ship a half-tagged document.
- **Cons**: A single recalcitrant article blocks the entire batch. Most heuristic failures are recoverable downstream by Markup or by the operator. The pipeline already uses `RuleSeverity.Optional` for the existing author rules precisely because aborting is too aggressive for this domain.
- **Why rejected**: Pipeline orchestration today (`FormattingPipeline.Run` in `DocFormatter.Core/Pipeline/`) is built around skip-friendly rules. Switching to abort-on-failure semantics for Phase 2 would either fragment the policy across rules or force the same change on Phase 1.

## Consequences

### Positive

- Consistent with existing Phase 1 rule severity (`Optional`).
- Operator still receives a clear signal in `diagnostic.json` about what was skipped.
- No risk of generating DTD-invalid markup from a confused rule.

### Negative

- A rule that skips silently (without a clear diagnostic message) creates user confusion. Mitigated by requiring structured warnings.
- A heuristic that under-detects (false negatives) is harder to spot than one that over-emits, because skipped tags do not surface in the document. The diff gate in ADR-003 mitigates this.

### Risks

- **Risk**: A rule skips so often that it provides no value.
  **Mitigation**: Each release's diff gate measures coverage on the 10-pair corpus; a rule that fails to mark too many cases blocks promotion until heuristics are improved.

## Implementation Notes

- Reuse the existing `IReport` and `diagnostic.json` infrastructure in `DocFormatter.Core/Reporting/`.
- Define reason codes per rule as constants (e.g., `static class HistRuleSkipReasons`).
- Severity for new Phase 2 rules: `Optional` (mirrors `ExtractAuthorsRule`).
