# ADR-004: Diagnostic JSON schema — per-field with confidence + issues list

## Status

Accepted

## Date

2026-05-05

## Context

The PRD specifies that a `<name>.diagnostic.json` file is generated **only** when the run logs at least one `[WARN]` or `[ERROR]`. The PRD listed the JSON schema as an Open Question. Without a locked schema, the implementation cannot start, and downstream consumers (today: humans scanning a folder; tomorrow: potential dashboards) have no contract to depend on.

## Decision

Lock the diagnostic JSON to a **two-part shape**: extracted field values with confidence flags, plus an ordered list of issues.

```json
{
  "file": "article.docx",
  "status": "warning",
  "extractedAt": "2026-05-05T18:30:00Z",
  "fields": {
    "doi": { "value": "10.1234/abc", "confidence": "high" },
    "elocation": { "value": "e2024001", "confidence": "high" },
    "title": { "value": "On the Behavior of...", "confidence": "high" },
    "authors": [
      {
        "name": "José Silva",
        "affiliationLabels": ["1"],
        "orcid": "0000-0002-1825-0097",
        "confidence": "high"
      },
      {
        "name": "Jane Doe Jr",
        "affiliationLabels": ["2"],
        "orcid": null,
        "confidence": "low"
      }
    ]
  },
  "issues": [
    { "rule": "ParseAuthors", "level": "warn", "message": "Author 2 ('Jane Doe Jr'): suspected comma in name, manual review suggested." }
  ]
}
```

Confidence is a closed enum: `high | medium | low | missing`. `status` is `ok | warning | error` and matches the highest severity in `issues`. `extractedAt` is UTC ISO-8601. Author records preserve insertion order. When a field could not be extracted, its `value` is `null` and `confidence` is `missing`.

The schema is serialized with `System.Text.Json` using camelCase. No JSON Schema file is generated for the MVP — the format is documented in the TechSpec and locked by integration tests in Phase 2.

## Alternatives Considered

### Alternative 1: Plain — issues list only

- **Description**: `{ file, status, issues: [...] }`. Mirror `report.txt` in JSON form.
- **Pros**: Trivial to serialize. Smallest schema surface.
- **Cons**: Editor still has to open the `.docx` to see what was extracted. The MVP's whole point is to surface uncertainty per field — without the per-field block, the editor cannot decide whether the warning is acceptable or needs intervention.
- **Why rejected**: User explicitly chose option B for the diagnostic schema.

### Alternative 2: Full FormattingContext mirror

- **Description**: Serialize the entire `FormattingContext` plus issues. Includes fields the MVP does not use (e.g., internal article ID, history block placeholders) so the schema is forward-compatible with future rules.
- **Pros**: One serializer to maintain across phases.
- **Cons**: Couples the diagnostic file to internal model layout. Adds noise in the MVP. Adding a Phase 2 field changes the schema and breaks consumers.
- **Why rejected**: Diagnostic file is a contract for editors, not an internal dump. Forward-compatibility is solved by adding fields to a stable shape, not by exposing internals.

## Consequences

### Positive

- Editors can scan a batch and answer "which articles have which fields uncertain" without opening every file.
- Future consumers (dashboard, queue, audit script) have a stable contract.
- Confidence enum is small and reviewable; new values require an explicit decision.

### Negative

- Two structures to keep in sync (`FormattingContext` → diagnostic field block) when adding a new extracted field.
- `confidence` semantics need to be explicit per rule (what makes a parse "low" vs "medium" must be documented in each rule).

### Risks

- **Risk**: Confidence becomes meaningless if rules treat it inconsistently (e.g., some never emit `low`).
  - **Mitigation**: Each rule's severity-to-confidence mapping is documented in the TechSpec and reviewed in code review.
- **Risk**: Schema drift between MVP and Phase 2.
  - **Mitigation**: Adding fields is allowed (consumers must ignore unknown). Renaming or removing fields requires a new ADR.

## Implementation Notes

- Confidence assignment for the MVP rules:
  - **DOI / ELOCATION**: `high` if extracted from the top table; `missing` if absent. Never `low` or `medium` in the MVP — the table is the only source.
  - **Title**: `high` if line 3 of the input is non-empty; `missing` otherwise.
  - **Authors**: `high` if every superscript label has a corresponding affiliation label downstream and the split produced no suspicious fragments; `low` if either check fails; `missing` if the authors paragraph could not be located.
- The diagnostic file is written next to the output `.docx` inside `formatted/`. File naming: `<inputBaseName>.diagnostic.json`.
- When `status` is `ok` (no warnings or errors), the file is **not** written — the `report.txt` is the sole record of a successful run.
