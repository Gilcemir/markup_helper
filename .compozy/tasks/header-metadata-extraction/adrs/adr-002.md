# ADR-002: Solution layout — Core + Cli + Tests, no Gui in MVP

## Status

Accepted

## Date

2026-05-05

## Context

The original DocFormatter spec (`instructions.md`) defines a 4-project solution: `DocFormatter.Core`, `DocFormatter.Cli`, `DocFormatter.Gui` (Avalonia), and `DocFormatter.Tests`. The PRD scopes the MVP to a CLI-only delivery with the GUI deferred to Phase 3, and the success criterion is "1 production article runs end-to-end." The Tests project is required because the MVP includes unit tests for the author parser (the most error-prone rule).

Three layout options exist:
1. Two projects (Core + Cli) — minimal, drop Tests until Phase 2.
2. Three projects (Core + Cli + Tests) — drop only the GUI.
3. Four projects with Gui as an empty stub.

## Decision

Use **three projects from day 1**: `DocFormatter.Core` (pipeline + rules + models), `DocFormatter.Cli` (entry point + CLI argument parsing + filesystem orchestration), and `DocFormatter.Tests` (xUnit, scoped to the author parser). Do **not** create `DocFormatter.Gui` in the MVP; it will be added in Phase 3 of the PRD's rollout plan.

All three projects target `net10.0` (per ADR-005, which overrides the spec's `net8.0` default). The MVP solution file `DocFormatter.sln` only references the three projects.

## Alternatives Considered

### Alternative 1: Core + Cli only (no Tests)

- **Description**: Drop the Tests project until Phase 2. Validation is fully manual using one production article.
- **Pros**: One less csproj to maintain; faster initial setup.
- **Cons**: Author parsing is the highest-risk rule per the PRD; without unit tests covering the named edge cases (`, Jr.`, names containing "and", absent ORCID, multiple affiliation labels), regressions can slip through silently. The PRD explicitly chose option B for testing scope ("unit tests só do parser de autores").
- **Why rejected**: Conflicts with the PRD's testing-scope decision.

### Alternative 2: Four projects with Gui as empty stub

- **Description**: Include `DocFormatter.Gui` as a stub csproj with `App.axaml` and `MainWindow.axaml` placeholders so the reference tree is in place from day 1.
- **Pros**: No refactor needed when Phase 3 starts.
- **Cons**: Carries Avalonia NuGet dependencies (`Avalonia`, `Avalonia.Desktop`, `Avalonia.Themes.Fluent`) for code that does nothing. Increases build time and the published `.exe` size if accidentally referenced. YAGNI violation.
- **Why rejected**: Adding the GUI project later is a one-time cost (~30 minutes); paying it now in dead dependencies and complexity is not justified by the saving.

## Consequences

### Positive

- The pipeline architecture (`FormattingPipeline`, `FormattingContext`, `Report`) is established in `DocFormatter.Core` from day 1, so Phase 2 rules slot in without refactor.
- The Tests project enforces the testing-scope decision from the PRD.
- The MVP build is minimal: no Avalonia dependencies, faster `dotnet publish`, smaller `.exe`.

### Negative

- Phase 3 will add `DocFormatter.Gui` and re-wire the solution file. Estimated ~30 minutes of mechanical work.
- A future contributor reading the solution might wonder why GUI is missing and need to consult this ADR.

### Risks

- **Risk**: The Cli project starts to absorb logic that belongs in Core because there is no GUI consumer pulling logic out.
  - **Mitigation**: Code review checklist — Cli holds only argument parsing, filesystem traversal, and process exit codes. All `.docx` manipulation lives in Core.

## Implementation Notes

- File layout under `DocFormatter.Core/`: `Pipeline/`, `Rules/`, `Options/`, `Models/`.
- File layout under `DocFormatter.Cli/`: `Program.cs` only for the MVP.
- File layout under `DocFormatter.Tests/`: one test class per parser, fixture `.docx` files in `Fixtures/Authors/`.
- `Directory.Build.props` at the solution root sets the build properties; see ADR-005 for the exact list.
